# Business Rules — Knowledge Base

| Code | Rule |
|---|---|
| BR-KB-001 | آیین‌نامه، بخشنامه و روال از نظر فرآیند ثبت، تأیید، انتشار، ویرایش و آرشیو یکسان هستند. |
| BR-KB-002 | سند رسمی ثبت‌شده توسط مدیر مستقیم منتشر می‌شود. |
| BR-KB-003 | سند رسمی ثبت‌شده توسط سرپرست تا قبل از تأیید مدیر برای کاربران عادی قابل مشاهده نیست. |
| BR-KB-004 | ویرایش سند توسط سرپرست نیازمند تأیید مدیر است. |
| BR-KB-005 | ویرایش سند توسط مدیر مستقیم اعمال می‌شود. |
| BR-KB-006 | ویرایش عادی سند نسخه جدید محسوب نمی‌شود. |
| BR-KB-007 | نسخه جدید فقط با عملیات مستقل «افزودن نسخه جدید» ایجاد می‌شود. |
| BR-KB-008 | بعد از انتشار نسخه جدید، نسخه قبلی حذف نمی‌شود و با وضعیت قدیمی/آرشیوی باقی می‌ماند. |
| BR-KB-009 | همه نقش‌ها می‌توانند نسخه‌های قدیمی را مشاهده کنند. |
| BR-KB-010 | نسخه‌های قدیمی باید با هشدار واضح نمایش داده شوند. |
| BR-KB-011 | هر سند باید حداقل به یک موضوع یا زیرموضوع وصل باشد. |
| BR-KB-012 | هر سند می‌تواند به چند موضوع یا زیرموضوع وصل شود. |
| BR-KB-013 | پیام‌های روزانه نیازمند تأیید مدیر نیستند. |
| BR-KB-014 | پیام‌های روزانه برای همه کاربران قابل مشاهده‌اند. |
| BR-KB-015 | تاریخ و ساعت ارسال پیام روزانه باید ذخیره شود. |
| BR-KB-016 | حذف فیزیکی سند مجاز نیست؛ سند فقط آرشیو می‌شود. |
| BR-KB-017 | همه کاربران می‌توانند برای اسناد جاری و قدیمی دکمه «مطالعه شد» را ثبت کنند. |
| BR-KB-018 | سیستم باید ثبت کند هر کاربر کدام سند، کدام نسخه و در چه زمانی مطالعه کرده است. |
| BR-KB-019 | متن داخل سیستم منبع اصلی محتوا است و فایل‌ها نقش پیوست دارند. |
| BR-KB-020 | تاریخ ثبت، تاریخ صدور، تاریخ انتشار، تاریخ آخرین تغییر و در صورت نیاز تاریخ اعتبار باید نگهداری شود. |
| BR-KB-021 | محتوای تأییدنشده نباید در API عمومی نرم‌افزار اصلی برگردد. |
| BR-KB-022 | در نتایج جستجو، نسخه‌های جاری باید نسبت به نسخه‌های قدیمی اولویت نمایش داشته باشند. |
| BR-KB-023 | برای رد درخواست تأیید، ثبت علت رد الزامی است. |
